## 2. background

- YOSHILOVER は半自動マーケ運用フェーズにあり、公開通知メールを Gmail 上で短時間に捌けることが重要。
- X 投稿は当面手動運用であり、メール本文の候補から人間が選んで投稿する前提を維持する。
- 通知 mail は `219` で件名 prefix 6 class に分類済みで、一覧上の視認性は既に上げる前提。
- user は疲労がある前提で、Gmail 側では「見落とさない通知」と「見なくてよい通知」の分離を行う。
- ChatGPT 側では、Gmail に溜まった候補を整理し、未処理抽出や優先順位づけを補助する。

## 3. Gmail ラベル設計

| ラベル | 推奨色 | 用途 | 備考 |
|---|---|---|---|
| `YOSHILOVER/投稿候補` | 青 | まず見る候補、X 手動投稿の主導線 | action 必要、目立たせる |
| `YOSHILOVER/要確認` | 黄 | 迷いが残る候補、後で判断 | 注意喚起 |
| `YOSHILOVER/警告` | 赤 | 緊急対応や即確認が必要な通知 | 最優先 |
| `YOSHILOVER/公開済` | グレー | 参照だけでよい完了通知 | 受信トレイをスキップ候補 |
| `YOSHILOVER/まとめ` | 紫 | daily / digest 系の集約通知 | 夜にまとめて見る |
| `YOSHILOVER/処理済み` | 緑 | 投稿済みや対応済みのマーキング | 完了管理用 |

- 色設定は `MKT-002` ratify 待ちだが、本便では上記を推奨値とする。
- `YOSHILOVER/` prefix を付けることで、個人メールや他 automation 系ラベルと混ざらない構造にする。
- `処理済み` は通知種類ではなく運用状態ラベルとして使い、`投稿候補` と直交で管理する。

## 4. Gmail フィルタ設計

各 filter は Gmail 設定の「フィルタを作成」UI から手動設定する。本便では Gmail filter の自動作成や一括変更は行わない。

| 条件 | アクション | 通知 |
|---|---|---|
| 件名に `【投稿候補】` | `YOSHILOVER/投稿候補` ラベル付与 | ON 候補 |
| 件名に `【要確認】` | `YOSHILOVER/要確認` ラベル付与 | 状況次第 |
| 件名に `【警告】` | `YOSHILOVER/警告` ラベル付与 + 重要マーク | ON 強め |
| 件名に `【公開済】` | `YOSHILOVER/公開済` ラベル付与 + 受信トレイをスキップ | OFF / 弱め |
| 件名に `【まとめ】` | `YOSHILOVER/まとめ` ラベル付与 + 受信トレイをスキップ | OFF / 弱め |
| 件名に `【緊急】` | `YOSHILOVER/警告` ラベル付与 + 重要マーク + 通知強め | ON 強め |

- `【警告】` と `【緊急】` はどちらも赤系の `YOSHILOVER/警告` に寄せ、通知ルールを単純化する。
- `【公開済】` と `【まとめ】` は Inbox から外し、スマホ通知疲れを増やさない。
- `【要確認】` は疲労度に応じて通知を切り替えられるよう、通知戦略を固定し切らない。

## 5. スマホ通知ルール

通知 ON:

- `【投稿候補】`
- `【警告】`
- `【緊急】`

通知 OFF / 低優先:

- `【公開済】`。件数が増えやすく、受信トレイをスキップして参照専用にする。
- `【まとめ】`。daily 集約前提なので、夜にまとめて確認する。

通知 ON / OFF 選択:

- `【要確認】`。疲労時は OFF、余裕がある日は ON を推奨する。

- 運用の主眼は「重要な候補だけ気付く」であり、全通知をリアルタイムに追わない。
- 通知ルールは user の生活リズムに合わせて微調整可能だが、`警告/緊急` を切らないことを原則にする。

## 6. ChatGPT triage 運用

| 文面 | 何をする |
|---|---|
| 「今日の投稿候補を一覧」 | `YOSHILOVER/投稿候補` ラベル + 当日受信 + 未処理を list 化する |
| 「未処理の投稿候補だけ出して」 | `YOSHILOVER/投稿候補` から `処理済み` を除いた差分だけ出す |
| 「警告メールある?」 | `YOSHILOVER/警告` ラベルの直近 24h を確認する |
| 「投稿済みにした候補を処理済みに」 | `YOSHILOVER/投稿候補` から `処理済み` へ状態移動する前提で整理する |
| 「今日投稿すべき上位 3 件を選ぶ」 | 投稿候補から記事 subtype と freshness を見て上位 3 件を抽出する |
| 「投稿候補の文面を見せて」 | mail 本文の `x_post_1` / `x_post_2` / `x_post_3` を表示する |

- ChatGPT は Gmail 上の一覧整理、候補抽出、本文比較の補助役とする。
- 手動投稿の実行そのものは user が X compose 画面で行う。
- `処理済み` 変更を ChatGPT に依頼する場合も、実際の Gmail 連携経路が使えるかは user 環境で最終確認が必要。

## 7. 運用手順

朝(起床直後):

- スマホ通知で `【警告】` と `【投稿候補】` の有無だけ確認する。
- `【警告】` があれば先に対処する。
- 警告がなければ `投稿候補` から朝の 1 件を選んで投稿する。

日中(スキマ時間):

- 受信トレイまたは `YOSHILOVER/投稿候補` ラベルを開く。
- 対象メール内の X intent URL をタップし、X compose 画面へ遷移する。
- 投稿後は ChatGPT に「処理済みに」と依頼するか、Gmail 側で手動ラベル変更を行う。

夜(寝る前):

- `YOSHILOVER/まとめ` を見て daily 状況を把握する。
- 余裕があれば `YOSHILOVER/要確認` を見て翌日に持ち越さない候補を減らす。
- 翌朝の優先候補を ChatGPT に 3 件選ばせ、朝の判断コストを下げる。

## 8. 投稿後のラベル変更

- 手動投稿後は Gmail UI で `YOSHILOVER/処理済み` を追加し、`YOSHILOVER/投稿候補` を外す運用を基本とする。
- `投稿候補` を残したい場合は、`処理済み` ラベルを追加した上で、一覧上は `処理済み` で除外検索する。
- ChatGPT 経由なら「投稿済みにした候補を処理済みに」と依頼して整理対象を明示する。
- filter での自動切替は本便スコープ外とし、将来の IMAP / Gmail API integration を `MKT-008` 候補として扱う。
