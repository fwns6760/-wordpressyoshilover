# Codex 依頼書 — 2026-04-18（第16便・1時間おき fact-check メール + 運用サマリ追加）

**依頼者**: よしひろさん
**前便**: `codex_responses/2026-04-18_15.md`（Scheduler 30分おき化、完了済 or 進行中）
**本便の目的**: Scheduler を **1時間おき** に再調整し、**0件窓はメール送信 skip** にしてスパム扱いを回避。同時に T-018 運用サマリを本文に追加

## 背景

- 第15便で Scheduler を `*/30 * * * *`（30分おき）に変更
- よしひろさん判断: **1時間に1通ペースが現実的**（30分は頻度過多でスパム扱いリスク）
- 方針:
  - Scheduler は `0 * * * *`（毎時 00 分）に再調整
  - 各 run で「直近1時間の新規/更新記事」と「直近24hのサマリ」を集計
  - **変化ゼロ AND 🔴ゼロ なら送信 skip**（平時 10-15通/日想定）
  - 🔴検出時は必ず送信

## 送信判定ロジック

```
1. 毎時 00分 scheduler 発火で /fact_check_notify?since=yesterday を実行
2. 直近1時間窓（modified が [now-1h, now]）の posts を抽出
3. メール送信判定:
   - 1時間窓に post 変化あり（新規 or modified） → 送信
   - OR 全体に 🔴 finding あり → 送信（繰り返し通知になっても red は重要）
   - どちらもなし → skip（fact_check_email_skipped ログのみ）
4. 送信時は 1通のメールに:
   - 件名: ヨシラバー MM/DD HH:00 🔴N 🟡M ✅K
   - 本文: 直近24h 全体状態 + 運用サマリ（T-018）
```

   def _should_send_email(reports, posts_in_last_hour) -> tuple[bool, str]:
       """
       Returns: (should_send, reason)

       - reports に red finding が1件以上 → True, "red_present"
       - どちらもなし → False, "no_change_no_red"
       """

3. **`run_notification()` を刷新**:
   - 上記 helper で判定
   - 送信する → 既存の send_email + `fact_check_email_sent` log（`reason` / `posts_in_last_hour_count` 追加）
   - skip → `fact_check_email_skipped` log（reason のみ）

4. **件名フォーマット変更**:
   - 既存: `ヨシラバー 事実チェック結果 MM/DD`
   - 新: `ヨシラバー MM/DD HH:00 🔴N 🟡M ✅K`
     - N, M, K は直近24h全体のカウント
     - HH は scheduler 発火時刻（JST）

5. **本文に運用サマリ追加**（T-018、全メール共通）:
   - Cloud Logging から直近24h の `rss_fetcher_run_summary` を集計
   - 集計項目:
     - `drafts_created` 合計
     - `created_subtype_counts` merge
     - `skip_duplicate` / `skip_filter` / `error_count` 合計
     - `x_post_count` 合計
     - publish カウント: 既存 log に該当フィールドあれば採用、無ければ `0件（Phase C 未解放）` 表示
   - 挿入位置: サマリ行（🔴N/🟡M/✅K/checked=X）直後、🔴 セクション前
   - フォーマット:
     ```
     📊 直近24h運用サマリ
     ─────────────
     📥 作成: 12件（lineup=5, postgame=3, manager=2, player=2）
     📤 公開: 0件（Phase C 未解放）
     🔄 スキップ: 重複=32 / フィルタ=99 / エラー=0
     🐦 Xポスト: 10件
     ```
   - 集計取得失敗時は「集計取得失敗」セクションで reason のみ表示、メール本体は送る

6. **test 追加**（`tests/test_fact_check_notifier.py`）:
   - `_should_send_email` の3ケース（window posts あり / red あり / どちらもなし）
   - 件名が `HH:00` 形式 + 件数サマリを含むこと（JST 時刻 mock で固定）
   - 運用サマリ取得 mock（`rss_fetcher_run_summary` 集計）が期待通り動くこと
   - 集計取得失敗 fallback でメール送信が止まらないこと

7. **env 変更禁止**。第14便の観測性（Message-ID / refused / smtp_response）は維持。
