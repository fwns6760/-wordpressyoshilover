## Step 1: 各ジョブの最終発火履歴（直近 48h）

### コマンド例

```bash
# ジョブごとに最後の発火時刻と結果を見る
for JOB in $(gcloud scheduler jobs list --location asia-northeast1 --project baseballsite --format='value(name.basename())'); do
  echo "=== $JOB ==="
  gcloud logging read \
    "resource.type=\"cloud_scheduler_job\" AND resource.labels.job_id=\"${JOB}\"" \
    --project baseballsite --freshness=48h --limit 20 --order=asc \
    --format='table(timestamp,severity,jsonPayload.status,jsonPayload.targetType)'
done
```

### 期待出力

- 各ジョブの直近 48h 発火タイムスタンプ列
- 成功 / 失敗 / retries の分布

### 成功判定

- 48h 分の発火履歴が全ジョブについて列挙されている
- 失敗が出ていれば severity と理由が見えている

### 失敗時

- ログ量が多すぎて切れる → limit を `100` に拡大、あるいは一部ジョブだけ深掘り
- scheduler ログが出ていない → 該当ジョブが disabled の可能性、`describe` で state を確認

### 所要時間

- 5 分

## Step 2: `/run` 側の flow summary 時刻分布（48〜72h）

### コマンド例

```bash
gcloud logging read \
  'resource.type="cloud_run_revision" AND resource.labels.service_name="yoshilover-fetcher" AND jsonPayload.event="rss_fetcher_flow_summary"' \
  --project baseballsite --freshness=72h --limit 200 --order=asc --format=json \
  | jq -r '.[] | [.timestamp, (.jsonPayload.drafts_created // 0), (.jsonPayload.publish_count // 0), (.jsonPayload.revision_name // "")] | @tsv'
```

### 期待出力

- 48〜72h の全 flow summary

### 成功判定

- timestamp が昇順で揃っている

### 失敗時

- 0 件 → `rss_fetcher_flow_summary` 以外のイベント名で回っている可能性、`jsonPayload.event` を grep して別名称を探す
- 72h で 0 件は異常 → severity ERROR を `freshness=72h limit=50` で取って原因を別途報告

## Step 3: 時刻帯別の集計表（Asia/Tokyo 0-23 時）

### コマンド例

```bash
# Step 2 の TSV を JST 時刻でバケット集計
gcloud logging read \
  'resource.type="cloud_run_revision" AND resource.labels.service_name="yoshilover-fetcher" AND jsonPayload.event="rss_fetcher_flow_summary"' \
  --project baseballsite --freshness=72h --limit 500 --order=asc --format=json \
  | jq -r '
      .[] |
      [
        ((.timestamp | sub("\\.\\d+Z$"; "Z") | fromdateiso8601) + 9*3600 | strftime("%Y-%m-%d %H:00 JST")),
        (.jsonPayload.drafts_created // 0),
        (.jsonPayload.publish_count // 0)
      ] | @tsv
    ' \
  | awk -F'\t' '{hr[$1]+=1; d[$1]+=$2; p[$1]+=$3} END {for (h in hr) print h"\t"hr[h]"\t"d[h]"\t"p[h]}' \
  | sort
```

### 期待出力

| 日時 (JST, 1h bucket) | flow 発火回数 | drafts_created 合計 | publish_count 合計 |
|---|---|---|---|

### 成功判定

- 72h 分のバケットが 60〜72 個（毎時 1 以上発火している前提）
- 18 時前後（17:00 / 18:00 / 19:00）のバケットが **他時間帯と比較して**有意に低いか判明

### 失敗時

- jq の timezone 変換が面倒 → Python で小スクリプト書いて bucket 集計（`datetime` + `pytz`、ただし `.env` の shell source は禁止）

### 所要時間

- 8 分

## Step 4: 18 時帯の個別追跡（Asia/Tokyo 17:00〜19:00）

### コマンド例

```bash
# 昨日 (2026-04-17) と今日 (2026-04-18) の 17:00-19:00 JST 範囲を個別に
gcloud logging read \
  'resource.type="cloud_run_revision" AND resource.labels.service_name="yoshilover-fetcher" AND (jsonPayload.event="rss_fetcher_flow_summary" OR jsonPayload.event="rss_fetcher_run_summary" OR severity>="WARNING")' \
  --project baseballsite --freshness=48h --limit 200 --order=asc --format=json \
  | jq '.[] | select((.timestamp | sub("\\.\\d+Z$"; "Z") | fromdateiso8601) as $t | (($t + 9*3600) % 86400) as $s | $s >= 17*3600 and $s < 19*3600)'
```

### 期待出力

- 17:00-19:00 JST の flow summary / run summary / WARNING+
- 発火回数、各回の drafts_created、エラー有無

### 成功判定

- 少なくとも 2 日（04-17 / 04-18）の 17-19 時帯が観測されている
- 発火 0 件の時間帯 / drafts_created=0 の時間帯が明示されている

### 失敗時

- jq timezone 処理が面倒 → awk で代替 or Python スクリプトに落とす

### 所要時間

- 5 分
