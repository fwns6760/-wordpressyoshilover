## やること

### 1. canary revision の build + deploy

```bash
gcloud run deploy yoshilover-fetcher \
  --source . \
  --region asia-northeast1 \
  --project baseballsite \
  --update-env-vars ARTICLE_INJECT_TEAM_STATS=1 \
  --no-traffic \
  --tag canary-team-stats
```

- `--no-traffic`: 新 revision は traffic 0% で生まれる（prod は 00139-k7h を維持）
- `--tag canary-team-stats`: tag 付き専用 URL を発行
- `--update-env-vars` は **追加だけ**（`--set-env-vars` 全量上書きは既存 env 破壊の恐れ、絶対禁止）
- `.codex` / `data/` は `.gcloudignore` で除外されている前提（第28-31便と同じビルド設定）。除外されていない場合でも、本便ではこれらを含めず source を build できる状態のみ確認し、`.gcloudignore` 自体の編集は行わない

### 2. describe で状態確認

```bash
gcloud run services describe yoshilover-fetcher --region asia-northeast1 --project baseballsite
```

確認項目（response doc に抜粋を貼る）:

- `LATEST` revision = canary（新 revision 名、`yoshilover-fetcher-00140-...` 想定）
- `traffic`:
  - `yoshilover-fetcher-00139-k7h` が **100%** であること
  - canary が **0%** であること
  - canary に `tag: canary-team-stats` が付いていること
- canary revision URL（`https://canary-team-stats---yoshilover-fetcher-...run.app` の形）
- canary revision の env に `ARTICLE_INJECT_TEAM_STATS=1` が含まれていること
- prod (`00139-k7h`) の env に `ARTICLE_INJECT_TEAM_STATS` が **依然として存在しないこと**（OFF 担保）
- 既存 env（`RUN_DRAFT_ONLY=0`, `AUTO_TWEET_ENABLED=0`, `PUBLISH_REQUIRE_IMAGE=1`, `ENABLE_PUBLISH_FOR_POSTGAME=1`, `ENABLE_PUBLISH_FOR_LINEUP=1`, `GMAIL_APP_PASSWORD_SECRET_NAME=yoshilover-gmail-app-password` ほか）が canary 側でも維持されていること

### 3. tag URL に直接 POST して smoke

```bash
CANARY_URL="$(gcloud run services describe yoshilover-fetcher \
  --region asia-northeast1 --project baseballsite \
  --format='value(status.traffic[?tag=`canary-team-stats`].url)')"

curl -i -X POST \
  -H "Authorization: Bearer $(gcloud auth print-identity-token)" \
  "${CANARY_URL}/run"
```

- レスポンス HTTP / latency / body を記録
- canary URL が空で取れた場合は `gcloud run services describe ... --format=yaml` 全文から手で URL を拾う

### 4. Cloud Run log 観測（canary revision 限定）

filter 例:

```
resource.type="cloud_run_revision"
resource.labels.service_name="yoshilover-fetcher"
resource.labels.revision_name="<canary revision 名>"
```

抜き取る項目:

- `/run` HTTP 200
- `rss_fetcher_run_summary` の `drafts_created` / `error_count` / `skip_filter` / `skip_duplicate`
- `rss_fetcher_flow_summary` の `prepared_subtype_counts` / `created_subtype_counts`（`postgame` / `lineup` の有無）
- `article_team_stats_injected` イベント件数 + 各イベントの `category` / `line_count`
  - **期待**: 試合速報 or 首脳陣 の draft が 1 本以上できた場合は **1 件以上**出る
  - 試合速報 / 首脳陣 が 0 件だった場合（カテゴリ全部が他カテゴリ）は **0 件**でも OK（その旨明記）
- `team batting stats 取得失敗` warning の有無（出ていたら原文ログを response doc に貼る）
- 既存 `article_ai_route_override` / `Gemini strict fact mode` 系ログが第31便と同レンジ

### 5. WP REST 確認

canary 経由で生成された draft（`drafts_created ≥ 1` の場合）の post_id を log から拾い、最大 2 本まで `GET /wp-json/wp/v2/posts/<id>?context=edit` で取得。各 post について以下を確認:

- `category`（試合速報 / 首脳陣 / その他）
- 本文に「【参考：巨人打者の今季主要指標」が含まれるか
  - **試合速報 or 首脳陣 → 含まれる** が期待
  - **その他カテゴリ → 含まれない** が期待
- 第1弾の `<div class="yoshilover-related-posts">` が候補ありの post で出ていること（第1弾回帰なし）
- 本文末尾の構造が壊れていないこと（`</div>` 閉じ忘れ等の HTML 異常がない）

source-fact 矛盾の判定は本便スコープ外（fact_check メールが拾う）。本便は **prompt に block が反映されていることの痕跡**だけ確認する。

### 6. 異常時 rollback 手順

以下のいずれかを観測した場合、**canary を即削除**（traffic は元から 0% なので prod 影響はないが、誤クリック promote リスクを潰す）:

- canary `/run` が HTTP 5xx / `error_count > 0`
- prod traffic が 100% から動いた（describe で 00139-k7h ≠ 100%）
- prod revision の env に `ARTICLE_INJECT_TEAM_STATS` が混入した（=コマンド事故）
- canary log で「team batting stats 取得失敗」が連発（>= 3 回）
- `article_team_stats_injected` が試合速報 / 首脳陣 **以外**の category で出た（仕様違反）
- 第1弾 `yoshilover-related-posts` div が候補ありの draft で消失した（第1弾回帰）

rollback コマンド:

```bash
# canary tag を外す
gcloud run services update-traffic yoshilover-fetcher \
  --region asia-northeast1 --project baseballsite \
  --remove-tags canary-team-stats

# canary revision を削除（prod=00139-k7h は不変）
gcloud run revisions delete <canary revision 名> \
  --region asia-northeast1 --project baseballsite --quiet
```

rollback 実施後、再度 describe を取って prod が 00139-k7h 100% / canary 不在 を確認、response doc に貼る。
