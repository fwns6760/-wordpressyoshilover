## T-007 🔴 p=62518 の score 根拠が `25-97` になった原因

### 結論

`25-97` は試合スコアではなく、参照元ページの生テキストに含まれていた UUID 断片から
`_extract_score()` が誤抽出した値です。  
同時に、同じ記事では venue 側も NPB 月間日程ページの切り出し誤りで `PayPay` を誤取得しており、
`62518` の red 判定は **2つの fallback バグが同時に出た結果** です。

### 原因

1. `src/acceptance_fact_check.py:311-328`
   `_source_reference_facts()` が source URL の title / description / text を雑に連結し、
   その全文に対して `_extract_score()` をかけています。

2. `_extract_score()` は `(\d{1,2})[-－–](\d{1,2})` のような単純パターン抽出です。
   `62518` の source URL
   `https://baseball.yahoo.co.jp/npb/game/2021038726/top`
   の本文には UUID 断片 `...4725-97a1...` が含まれており、
   ここから `25-97` を score として誤認しました。

3. `src/acceptance_fact_check.py:356-375`
   `_fetch_npb_schedule_snapshot()` は NPB 月間日程ページを date 付近の文字数切り出しで処理しています。
   この切り出しがズレると別試合の会場を拾い、`62518` では `PayPay` が venue fallback になっていました。

4. `src/acceptance_fact_check.py:518-570`
   `_check_game_facts()` は postgame で
   `reference.get("score") or source_facts.get("score")`
   を使うため、live reference 側 score が空だと誤抽出した `25-97` が採用されます。

5. さらに evidence URL は `reference["evidence_urls"][0]` 固定なので、
   実際は source URL 由来の誤値なのに、レポート上は NPB / Yahoo game reference 側が根拠のように見えます。

### 影響範囲

#### 62518 で実際に表面化した異常

- `post_id=62518`
  - venue expected: `PayPay`
  - score expected: `25-97`

#### 同種の score 誤抽出を確認した post_id

- `62527`
  - source facts 側で `score=19-4` を誤抽出
- `62540`
  - source facts 側で `score=19-4` を誤抽出

上記 2 件は今回の run では absurd red としては表面化していませんが、
`source_facts.score` に依存する経路へ入ると同種事故になります。

#### 同じ fallback 系統の venue 誤取得が疑われる既知チケット

- `62044`
- `61981`
- `61886`
- `61802`
- `61770`
- `61598`

`61981` と `61598` は spot check で venue fallback の異常を再確認しました。

### 修正方針

実装は今回未着手ですが、修正の筋は次の通りです。

1. `source_facts` からの score 抽出を停止する
   - 少なくとも raw page text 全体への `_extract_score()` は禁止
   - score は structured source のみ許可する

2. `_fetch_npb_schedule_snapshot()` の date block 切り出しをやめ、
   月間日程 HTML の構造に沿って対象日行だけを抽出する

3. `_check_game_facts()` で expected 値の供給元に応じて evidence URL を分ける
   - `reference` 由来なら game reference URL
   - `source_facts` 由来なら source URL

4. `62518` のような case を regression test 化する

## T-001 🔴 WP REST API の `?status=draft` フィルタが効かない原因

### 結論

権限ではなく、WordPress 側のカスタムプラグイン
`src/yoshilover-exclude-cat.php`
が collection query を広くフックし、`post_status=publish` を強制しているのが原因です。

### 調査結果

#### 実 API 確認

- `GET /wp-json/wp/v2/posts?status=draft&per_page=5&context=edit`
- `GET /wp-json/wp/v2/posts?status=publish&per_page=5&context=edit`
- `GET /wp-json/wp/v2/posts?status=any&per_page=5&context=edit`

上記 3 つはすべて同じ publish 記事一覧を返しました。

一方で、

- `GET /wp-json/wp/v2/posts/62518?context=edit`

は `status=draft` を正しく返しています。

また、

- `GET /wp-json/wp/v2/users/me?context=edit`

は `200` で、認証ユーザーは draft を閲覧できる権限を持っていました。

#### 原因コード

`src/yoshilover-exclude-cat.php:16-44`

```php
add_action( 'pre_get_posts', function( WP_Query $query ) {
    if ( is_admin() ) {
        return;
    }
    ...
    $query->set( 'post_status', 'publish' );
}, 1 );
```

特に line 43 の

```php
$query->set( 'post_status', 'publish' );
```

が REST collection request にも効いていると考えられます。

この plugin は front page / feed 除外目的ですが、条件が広く、
REST の post collection query まで巻き込んで publish 強制になっています。

### 修正方針

実装は今回未着手です。

1. `REST_REQUEST` 中はこの plugin を効かせない
2. もしくは true front-end query のみに条件を絞る
3. `post_status=publish` の強制はテーマ用一覧 query のみに限定する

権限不足ではないため、修正対象は WP 側 query filter です。

## T-003 🟠 現在の draft 件数を subtype 別に取得する手段

### 選択した案

**B. Cloud Logging クエリで候補 post_id を拾い、WP 単独 GET で現在 draft を棚卸しする**

理由:

- `?status=draft` の list API が壊れていても使える
- production deploy が不要
- 現在の draft のみを最終的に単独 GET で確定できる

### 実装内容

追加ファイル:

- `src/draft_inventory_from_logs.py`
- `tests/test_draft_inventory_from_logs.py`

CLI:

```bash
python3 -m src.draft_inventory_from_logs --days 7
python3 -m src.draft_inventory_from_logs --days 7 --json
```

処理:

1. Cloud Logging から以下のログを収集
   - `[WP] 記事draft post_id=`
   - `[WP] 下書き作成 post_id=`
   - `[WP] 既存記事を再利用 post_id=`
2. 候補 `post_id` を重複排除
3. 各 `post_id` を `WPClient.get_post()` で単独取得
4. `status == draft` のものだけ残す
5. `draft_audit.audit_post()` で subtype / category を再判定
6. subtype 別件数を出力

### 現在の draft 件数

ローカル確認コマンド:

```bash

```

結果:

category / subtype 別:

- `ドラフト・育成/farm_lineup: 2`
- `補強・移籍/roster: 1`

- `試合速報/postgame: 9`
- `試合速報/pregame: 4`
- `選手情報/player: 11`

### 補足

この手段は「直近 N 日でログに出た draft 候補」ベースです。  
運用上の current inventory としては十分使えますが、
完全な DB 全量在庫照会ではありません。

list API 修正後は、この CLI を
- current inventory の簡易棚卸し
- 受け入れ試験前の件数確認
に使う想定です。
