## 5. 修正案 (a)(b)(c) の比較

### 前提件数

- 総対象: `19件`
- URL が本文に残っている: `14件` (`B=12`, `C=2`)
- URL が本文に残っていない: `5件` (`A=5`)

| 案 | 何をするか | 即時カバー見込み | 副作用リスク | 工数見積り | 評価 |
|---|---|---:|---|---|---|
| (a) | `extract_source_links()` の抽出パターン拡張 | **14/19件** | 低〜中。過剰抽出だけ注意 | 小〜中 | **第一優先** |
| (b) | 記事側に source block を後追い補填 | **14/19件は自動候補、5/19件は手作業前提** | 中〜高。公開記事改変が入る | 中〜大 | 第二段階向き |
| (c) | 古い記事を source_reference 検査対象外にする | **最大5/19件** を抑制可能 | 中。yellow を隠すだけで根治しない | 小 | 補助策に限定 |

### (a) `extract_source_links()` 拡張の内訳

想定追加パターン:
- `【引用元】` 見出し + 次段落の anchor
- `引用元:` / `出典:` / `参考:` / `source:` ラベル付き段落
- 本文末尾の灰色小文字 footer リンク
- `[[1]](url)` 形式

これで直接カバーできる見込み:
- `C=2` 全件
- `B=12` 全件
- **合計 `14件`**

### (b) 記事側補填の内訳

自動補填しやすい:
- `B=12`, `C=2` の **14件**
  - 既に本文に URL があるため backfill バッチ化の余地あり

補填が難しい:
- `A=5`
  - source URL 自体が HTML にない
  - 元 source の再探索か手動調査が必要

### (c) 除外ルールの実効範囲

現実的に除外候補となるのは:
- `A=5`

ただし:
- `B/C` は「記事側に source があるのに見えていない」ので除外すべきではない
- `post_id` や `日付` で一律除外すると、回収可能な 14件まで隠す副作用が大きい

## 6. 推奨方針

**推奨は 1 本に絞ると `(a) extract_source_links() の拡張を先にやる` です。**

理由:
1. **19件中 14件を、記事改変なしで即時に救える**
2. 公開記事そのものを触らず、監査ロジックだけで改善できる
3. `A=5` と `B/C=14` を切り分けたうえで、残課題を小さくできる

推奨順序:
1. 次便で `(a)` 実装
2. publish 36件へ再監査
3. その時点で残る `A=5` だけを
   - `(b)` source block 補填
   - もしくは `(c)` 旧記事例外
   のどちらで扱うか決める
