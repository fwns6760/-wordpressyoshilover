## 5. 画像処理ロジックの入口

主な入口:

- [src/rss_fetcher.py](/home/fwns6/code/wordpressyoshilover/src/rss_fetcher.py)
  - `fetch_article_images()`:
    - `og:image` を最優先で拾う
  - 記事生成フロー:
    - `_article_images = fetch_article_images(...)`
    - `_og_url = _article_images[0] if _article_images else ""`
    - `wp.upload_image_from_url(_og_url, source_url=post_url)`
- [src/wp_client.py](/home/fwns6/code/wordpressyoshilover/src/wp_client.py)
  - `upload_image_from_url()`:
    - 画像 URL を取得
    - `Content-Type` を判定
    - 許可 MIME でなければ `0` を返す
  - `_get_image_candidate_exclusion_reason()`:
    - URL ベースの早期除外
  - `_get_image_mime_exclusion_reason()`:
    - MIME ベースの除外

失敗時の挙動:

- upload に失敗しても例外で止めず `0` を返す
- draft 作成自体は継続
- `PUBLISH_REQUIRE_IMAGE=1` は publish 時のみ効くため、draft は画像なしで残る

## 7. 最有力仮説

最有力 1:

- `social_news` / X 由来記事で、画像候補の先頭に `https://abs.twimg.com/emoji/v2/svg/26a0.svg` が入っている
- `fetch_article_images()` は `og:image` 優先で先頭候補を採用する
- `wp.upload_image_from_url()` は SVG を拒否するため `featured_media=0` になる

補強材料:

- 62584 / 62585 / 62587 で **同じ SVG URL** が実ログに出ている
- 3 件とも `featured_media_observation_missing`

最有力 2:

- URL ベースの絵文字 SVG 除外が不完全
- 現行 [src/wp_client.py](/home/fwns6/code/wordpressyoshilover/src/wp_client.py) の `_get_image_candidate_exclusion_reason()` は `abs-0.twimg.com/emoji/` しか見ていない
- 実ログの URL は `abs.twimg.com/emoji/...` で、**事前除外をすり抜けて MIME 判定まで進んでいる**

## 8. 改善方針提案（本便では未実施）

優先度高:

1. **URL 除外条件の拡張**
   - `abs.twimg.com/emoji/` も `emoji_svg_url` として早期除外
2. **1件目失敗時の fallback**
   - 先頭画像が SVG/不正 MIME なら、`_article_images[1:]` へ再試行
3. **観測性の格上げ**
   - `featured_media_observation_missing` を INFO のままでも良いが、少なくとも `excluded_url` / `content_type` / `source_type` を見やすく残す

優先度中:

4. **ソース種別別 fallback**
   - `social_news` / X 由来は `og:image` を盲信せず、本文画像やプレースホルダへ fallback

WONTFIX 判断材料:

- 「62584 だけの単発」ではないので、完全 WONTFIX は弱い
- ただし全記事の画像パイプライン故障ではなく、**特定ソースパターン限定の局所不具合** と見るのが妥当
