## Step 5: subtype × 期待構成の叩き台

### postgame

- 結果 → 流れが動いた局面 → キーマン → 次戦への含み
- ヨシラバー固有: 同カード直近3試合や、同投手相手の打線傾向とつなぐ
- AI 活用: 直近5試合の得点推移、同カード別戦績、個人成績の即時比較
- 「どこで勝敗が傾いたか」を 1 イニング単位で固定する
- 「今日勝った/負けた」ではなく「次に何が残ったか」まで書く

### lineup

- 発表スタメン → 前回との差分 → 起用意図仮説 → 試合前の注目点
- ヨシラバー固有: 直近3〜5試合の打順変化、外れた選手・上がった選手の意味
- AI 活用: 相手先発との対戦傾向、左右・守備位置・最近打席結果の比較
- 「誰が出たか」だけでなく「どこが動いたか」を中心に置く
- 読者が試合開始直後に見るべき 1 点で締める

### manager

- 発言要旨 → どの序列/采配を動かす話か → 過去発言とのトーン差 → 次の起用論点
- ヨシラバー固有: 阿部監督の最近3本のコメントと今回の差を拾う
- AI 活用: 類似発言の時系列比較、前後のスタメン・継投との接続
- quote 要約ではなく「意味づけ」を主役にする
- 一番動くのが誰かを 1 人か 1 ポジションに絞る

### notice

- 公示要旨 → 対象選手の現状 → 背景 → チーム編成への影響
- ヨシラバー固有: 同ポジションの空席/競争相手と結びつける
- AI 活用: 直近一軍/二軍出場履歴、登録抹消の時系列整理
- 「登録/抹消」だけで終えず、編成の文脈を入れる
- 次の公示や試合でどこを見るべきかで締める

### pregame

- 試合前焦点 → 先発/打線/相手要素 → 今日の分岐点
- ヨシラバー固有: 前日の試合や直近カードからの課題継続を示す
- AI 活用: 相手先発との相性、直近5試合の得点分布、対戦成績
- 「勝てるか」ではなく「どこが試合の分かれ目か」を先に置く
- 読者が試合開始直後に確認したい1ポイントで終える

### recovery

- 故障/復帰要旨 → 現段階 → チーム影響 → 次の復帰マイルストーン
- ヨシラバー固有: その選手不在で誰が埋めているかを接続する
- AI 活用: 過去の離脱/復帰スケジュール比較、ポジション別の穴の大きさ
- 医療推測は避けつつ、起用文脈は厚くする
- 「順調です」より「次の節目は何か」を書く

### farm

- 二軍結果/活躍 → 数字 → 一軍への示唆 → 次の昇格論点
- ヨシラバー固有: その選手が今このタイミングで浮上する理由を書く
- AI 活用: 二軍成績の推移、同ポジション若手比較、支配下争い整理
- 二軍の単発結果で終えず、一軍接続を明示する
- 読者が「次の公示を見たくなる」締めにする

### social

- 話題要旨 → 発信内容 → 試合/選手文脈 → ファン視点の意味
- ヨシラバー固有: その post を巨人文脈に翻訳する役目を担う
- AI 活用: 動画内プレーの局面整理、前後プレーや最近の流れへの接続
- tweet 要約を最短で終えて、背景比率を増やす
- 「この post を記事化する意味」を1行で示せる構成にする

### player

- 現状整理 → 変化点/コメント → 今季文脈 → 次の注目
- ヨシラバー固有: 過去登板・過去打席・登録履歴から現在地を置く
- AI 活用: 成績推移、同ポジ比較、前回記事からの変化抽出
- 200〜350 字では足りないため、status 系でも最低 450 字は必要
- 「何が変わったか」を一言で言える構成にする

### general

- ニュース要旨 → 背景 → その話題が巨人ファンに刺さる理由 → 次の論点
- ヨシラバー固有: 巨人ファンが今この話題を気にする理由を1文で置く
- AI 活用: 広域参照で「今季の流れの中の位置」を付与する
- generic 要約に落ちやすいので、毎回テーマ軸を固定する
- 何を見ればいい記事かを冒頭で明示する

## Step 6: 結論と次便候補

### 1. 全体評価

現行 prompt は、**安全性・誤情報防止・構造の最低保証**という点ではかなり強いです。一方で、source facts に閉じた strict 制約と、低い min chars、fallback 優先の設計が組み合わさっているため、「他サイトより深い」「ヨシラバーだから読む意味がある」本文には伸びにくい状態です。subtype 分岐はあるものの、分岐の目的が「深くする」ではなく「ぼかさない」に寄っており、ヨシラバー × AI の掛け算はまだ入っていません。

### 2. 薄さの主因 top 3

1. **許容文字数が低すぎる**
   - `player_status 200-350`、`player_quote 300-400`、`social 380-680`
   - `_get_gemini_strict_min_chars()` は `160/220`
   - Step 4 でも本文コアは `290〜546` 字に収束

2. **prompt が source-only で、差別化の材料が禁止されている**
   - 「検索して新事実を足さない」「使ってよい事実だけ」
   - これ自体は安全だが、過去記事・今季推移・比較軸が消える
   - 結果として **他サイトの要約との差別化が弱い**

3. **HTML の厚みと本文の厚みが乖離している**
   - hero card / CTA / buttons / widgets で HTML は長くなる
   - 例: `62000` は HTML 4973 文字だが本文コア 290 字
   - 「見た目は記事、読後感は短報」になりやすい

### 3. 次便候補（A〜F）

- (A) prompt rule の段階拡張
  - `postgame` と `lineup` から先行し、「今季文脈」「同カード比較」「次戦への含み」を明示ルール化
- (B) `build_news_block()` の HTML テンプレに構成セクション導入
  - `今日の論点` `今季文脈` `次の注目` を固定枠にする
- (C) 元ソースの引用範囲拡大
  - 現在の `summary` 依存から、本文抜粋や複数 source_fact を厚くする
- (D) **過去記事への参照導線**
  - 同選手 / 同カード / 同論点の内部リンクを自動挿入
- (E) **数値・データソース自動付与**
  - 選手成績、直近5試合、対戦成績などを generation input に追加
- (F) general / social を別設計に切り出し
  - tweet digest を無理に記事化しない基準も含む

### 4. 優先度推奨

**推奨: (D) 過去記事への参照導線**

理由:

- 「ヨシラバーで書く意味」を一番増やせるのは、**このサイト自身の蓄積を今日の記事へ接続すること**だから
- prompt だけを強くしても、source-only 制約のままでは深さが出にくい
- 同選手 / 同カード / 同テーマの過去記事を 1〜2 本つなぐだけで、要約サイトから **“継続して追っている巨人特化サイト”** へ変わる
