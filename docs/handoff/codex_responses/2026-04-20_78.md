## 2. コード側ロジック要約

- `src/rss_fetcher.py` の事前集約は `_aggregate_duplicate_sources()` のみで、実際に merge しているのは lineup 系の `(published_day, "lineup")` グループだけ。farm / pregame / postgame には同日 merge 経路がない。
- `src/rss_fetcher.py` 本体は各 prepared item に対し `_rewrite_display_title_with_template()` で rewritten title を作り、そのまま `wp.create_post(draft_title, content, status="draft")` を呼ぶ。same-fire 内で `post_id` を覚えるローカルガードはない。
- `src/rss_fetcher.py` の title template は collision を起こしやすい。今回の farm は `farm_result_score -> "巨人二軍 {score} 結果のポイント"`、pregame は `game_pregame_generic -> "巨人戦 試合前にどこを見たいか"`、postgame は `game_postgame_generic -> "巨人戦 試合の流れを分けたポイント"` に落ちやすい。
- `src/wp_client.py` の `find_recent_post_by_title()` は「24 時間以内」「正規化タイトル一致」で既存 post を検索し、draft 作成時でも `draft / pending / future / auto-draft` を reusable とみなす。`create_post()` は hit すると `_reuse_existing_post()` に流し、`post_id` を返して新規 `POST` しない。
- `_reuse_existing_post()` が更新するのは `featured_media` / `categories` / `status` だけで、本文 `content` と `title` は更新しない。つまり今回の 2 回再利用は「同じ draft id を返した」のであって、same-fire の 2 本目が本文を書き戻した証拠はない。
- `persist_processed_entry_history()` は `published=True` または `draft_only` 理由のときしか `rewritten_title_norm:*` を保存しない。今回の fire は `publish_disabled_for_subtype` だけなので、1 本目の処理後も history に rewritten title が積まれず、2 本目で `title_collision_detected` が出ない。

## 4. 根本原因仮説

- 判定は `bug`。ただし壊れているのは「same_day_merge 実装」ではなく、「retry 防止用の title-only 再利用が、generic rewritten title と組み合わさって別 entry を同一 draft に誤結合する」点。
- 根拠は 3 つある。1 つ目は今回の fire で `POST` / `PUT(content)` / `same_day_merge` の証拠がなく、両方とも `[WP] 既存記事を再利用` だけだったこと。2 つ目は `_reuse_existing_post()` が本文を更新しないこと。3 つ目は 7 日 trend でも同じ generic title 群で再発していること。
- `設計通り` と言える部分は `WPClient.find_recent_post_by_title()` 自体の目的が retry 抑止である点だけ。だが現状の key が `normalized rewritten title + 24h` なので、distinct source を duplicate 扱いする副作用が大きすぎる。
- 疑わしい関数は 2 箇所に絞れる。`src/wp_client.py` の `find_recent_post_by_title()` / `create_post()` が title-only reuse を行い、`src/rss_fetcher.py` の `_rewrite_display_title_with_template()` が farm / pregame / postgame を generic title に畳みすぎている。補助要因として `persist_processed_entry_history()` が draft-disabled fire では rewritten title を保存しない。
- race condition の証拠は薄い。今回の 2 回は `13:03:33 JST` → `13:03:44 JST` の直列処理で、Cloud Run 側も単一ループで進んでいる。

## 5. 次に Claude が判断すべきこと

- これは「設計確認便」より「修正便 scope 草案」に寄せてよい。user が title-only reuse を明示的に維持したい場合だけ設計確認に戻す。
- 修正 scope の最小案は 3 点。`src/wp_client.py` の draft 再利用 key を title-only から狭める、または `rss_fetcher.py` 側で same-fire の distinct `source_url` に対して reuse を禁止する、さらに generic title template に最低 1 つの識別子を残す。
- 優先順位は `1. WP draft reuse 条件の是正`、`2. same-fire guard 追加`、`3. pregame/postgame/farm の generic title 緩和`。今回の `62795` は 1 と 2 だけでも止まる。
- 追加すべき test は少なくとも 3 本。`同一 fire・別 farm URL・同一 score -> post_id が分かれる`、`同一 fire・別 pregame URL・generic title -> post_id が分かれる`、`同一 source の再実行 retry -> 既存 draft reuse は維持`。
- Claude への引き継ぎ文面は「same_day_merge 調査」ではなく「title-only recent-post reuse が generic rewritten title collision を起こしているので、distinct source を同一 draft に吸着させない修正 scope を切る」で十分。
