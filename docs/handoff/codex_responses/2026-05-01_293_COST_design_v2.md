## 1. v2 で追加する判断

- `src/post_gen_validate.py` は存在しない。現行 validator 本体は `src/rss_fetcher.py` 内の `_evaluate_post_gen_validate(...)` に内包されている。
- preflight skip は現状 `logger.info()` のみで、**local ledger / GCS ledger / publish-notice scanner / mail subject** のどこにも接続されていない。
- `post_gen_validate` の通知 path は既に成立しているが、persisted JSONL row には `record_type` / `skip_layer` が未保存。scanner 側が定数注入で補っている。
- `cloud_run_persistence.py` に **post_gen_validate 専用 ledger stanza は存在しない**。producer append は `rss_fetcher.py`、scanner-side GCS restore は `publish_notice_scanner.py` で分担されている。
- したがって 293 は、`post_gen_validate_history.jsonl` 共用よりも **専用 `preflight_skip_history.jsonl` を parallel path で増設**する方が狭く、安全。

### 2.2 preflight skip event の現状出力 path

現状の preflight skip は次の 2 経路しか持たない。

1. `src/rss_fetcher.py:4602-4606`  
   `telemetry["cache_hit_reason"] = "preflight_skip"`

2. `src/gemini_preflight_gate.py:290-300`

ここに **存在しないもの**:

- GCS append
- publish-notice scanner の consumer
- dedupe key
- subject prefix
- queue log

つまり現状は「skip 自体は分かるが、mail 系には 1 行も接続されていない」。293 の core gap はここ。

### 2.3 `post_gen_validate` 実体と skip event schema

`src/post_gen_validate.py` は無い。現行実体は以下。

| file:line | 役割 |
|---|---|
| `src/rss_fetcher.py:7640-7740` | `_evaluate_post_gen_validate(...)`。fail axis / stop_reason を返す。 |
| `src/rss_fetcher.py:7742-7788` | `_log_article_skipped_post_gen_validate(...)`。logger payload を作り、history append helper を呼ぶ。 |
| `src/rss_fetcher.py:7848-7891` | `_record_post_gen_validate_skip_history(...)`。local JSONL append + GCS append。 |

現状 schema の実測:

- logger payload には `skip_layer=post_gen_validate` が入る (`src/rss_fetcher.py:7762-7777`)
- persisted JSONL row には `skip_layer` も `record_type` も入らない (`src/rss_fetcher.py:7861-7870`)
- scanner 側は `record_type` / `skip_layer` を row から読むのではなく、定数 `_POST_GEN_VALIDATE_RECORD_TYPE` / `_POST_GEN_VALIDATE_SKIP_LAYER` を request 化時に注入している (`src/publish_notice_scanner.py:1118-1137`)

ここから分かること:

- 293 で preflight ledger を作るときは、**最初から row 自体に `record_type` / `skip_layer` を保存**したほうが後方互換を保ちやすい。
- ついでに post_gen_validate row も forward-compatible に `record_type` / `skip_layer` を持たせると、scanner 共通 helper 抽出が容易。

### 2.4 scanner の parallel path 接合点

`src/publish_notice_scanner.py` の既存接合点はかなり明確。

| file:line | 役割 |
|---|---|
| `src/publish_notice_scanner.py:690-695` | dedupe key = `post_gen_validate:{source_url_hash}:{skip_reason}` |
| `src/publish_notice_scanner.py:1027-1212` | `scan_post_gen_validate_history(...)` 本体 |
| `src/publish_notice_scanner.py:1305-1346` | `scan()` からの統合点。guarded review の残 cap を post_gen_validate に渡す。 |

重要 detail:

- GCS restore は scanner-side で実施 (`src/publish_notice_scanner.py:1063-1068`)。
- queue log には `record_type`, `skip_layer`, `source_url_hash` を載せる (`src/publish_notice_scanner.py:1141-1154`)。
- **cap 到達時に cursor を進めない** (`src/publish_notice_scanner.py:1156-1166`, `1182-1191`)。
  - これは silent skip 防止の要。293 の preflight path も同じ cursor semantics に揃えるべき。

293 への示唆:

- `scan()` への差し込み位置は、既存 289 を壊さないため **guarded → post_gen_validate → preflight** の後段が安全。
- shared cap は `remaining_review_cap = max(0, review_cap - already_emitted)` を段階的に再計算する。
- history dedupe window は既存 24h (`_HISTORY_WINDOW`) をそのまま共有してよい。

### 2.5 `cloud_run_persistence.py` の実態

`cloud_run_persistence.py` は publish-notice job entrypoint の state staging helper だが、現状の責務は次だけ。

| file:line | 役割 |
|---|---|
| `src/cloud_run_persistence.py:20-31` | cursor/history/queue/old-candidate/guarded-history の default path 定義 |
| `src/cloud_run_persistence.py:318-352` | entrypoint。本体 runner 実行の前後で state を download/upload |
| `src/cloud_run_persistence.py:339-349` | `guarded_publish_history.jsonl` は download-only (`upload_on_exit=False`) |

read-only 結論:

- **post_gen_validate ledger 専用 stanza はここに無い。**
- post_gen_validate は「producer append = `rss_fetcher.py`」「scanner restore = `publish_notice_scanner.py`」の split model。
- entrypoint 側で近い template は、`guarded_publish_history.jsonl` の **download-only ledger + cursor persisted separately** という扱い。

293 にそのまま適用すると:

- preflight ledger 自体は publish-notice job が mutate しないので、upload は不要。
- entrypoint に足すなら `preflight_skip_history_cursor.txt` の persist と、必要なら `preflight_skip_history.jsonl` の download-only staging が自然。
- Claude draft の「entrypoint restores and uploads preflight_skip_ledger」は read-only 実態とは少しズレる。**upload 対象は cursor / history / queue で十分、ledger 本体は download-only が筋**。

## 4. ledger schema 詳細

### 4.1 推奨: 専用 `preflight_skip_history.jsonl`

```yaml
preflight_skip_history_jsonl:
  format: jsonl
  local_default_path: /tmp/pub004d/preflight_skip_history.jsonl
  scanner_fallback_path: logs/preflight_skip_history.jsonl
  cursor_path: /tmp/pub004d/preflight_skip_history_cursor.txt
  gcs_bucket: baseballsite-yoshilover-state
  gcs_object: preflight_skip/preflight_skip_history.jsonl
  content_type: application/x-ndjson
  dedupe_key_formula: "preflight_skip:{source_url_hash}:{skip_reason}"
  row_schema:
    ts:
      type: iso8601-jst
      required: true
      example: "2026-05-01T14:05:00+09:00"
    record_type:
      type: const
      required: true
      value: preflight_skip
    skip_layer:
      type: const
      required: true
      value: preflight
    source_url:
      type: string
      required: true
      example: "https://example.com/article-1"
    source_url_hash:
      type: string
      required: true
      note: "sha256(source_url)[:16] と現行 duplicate/hash helper に合わせる"
      example: "abc123def4567890"
    content_hash:
      type: string
      required: true
      note: "_gemini_text_with_cache(...) ですでに算出済み"
      example: "feedface12345678"
    source_title:
      type: string
      required: true
      note: "candidate_meta.title をそのまま保存。mail subject の first fallback"
    category:
      type: string
      required: true
      example: "試合速報"
    article_subtype:
      type: string
      required: true
      example: "postgame"
    source_name:
      type: string
      required: false
      example: "スポーツ報知"
    source_type:
      type: string
      required: false
      example: "news"
    published_at:
      type: iso8601
      required: false
    has_game:
      type: boolean
      required: false
    skip_reason:
      type: string
      required: true
      allowed_values:
        - existing_publish_same_source_url
        - placeholder_body
        - not_giants_related
        - live_update_target_disabled
        - farm_lineup_backlog_blocked
        - farm_result_age_exceeded
        - unofficial_source_only
        - expected_hard_stop_death_or_grave
    existing_publish_same_source_url:
      type: boolean
      required: false
      note: "reason が existing_publish_same_source_url の時の説明補助"
```

推奨理由:

- preflight は `generated_title` をまだ持たない。
- reason set が 282 固有 8 種で、289 の reason resolver と混ぜると helper が太る。
- scanner / queue / cursor を `post_gen_validate` と parallel に作れる。

### 4.2 `post_gen_validate_history.jsonl` との並列ルール

```yaml
post_gen_validate_history_jsonl_forward_compatible:
  current_row_fields:
    - ts
    - source_url
    - source_url_hash
    - source_title
    - generated_title
    - category
    - article_subtype
    - skip_reason
    - fail_axis
  recommended_added_fields:
    record_type:
      value: post_gen_validate
    skip_layer:
      value: post_gen_validate
  reader_compatibility:
    legacy_rows_without_record_type_or_skip_layer: accepted
    scanner_behavior: "missing 時は既存定数で補完"
```

要点:

- 293 で preflight row を新規設計するなら、post_gen_validate row もこの機会に future-proof 化する。
- ただし scanner は legacy row を継続で読めるようにする。

- preflight ledger 本体に対する upload / mv は呼ばれない

- object missing は `False` 扱いで続行、hard failure にしない

## 10. 282 already ON 時の追加 rollback(handoff 要点 4 反映)

§6 Phase A/B/C は **282 flag OFF 前提**の rollback。282 が既に ON の場合、preflight skip event は **fetcher で実発生中** で、Phase A 単独 rollback だと:

- publish-notice 側で visible 化が止まる
- ただし fetcher 側 preflight skip 自体は継続発生 → silent skip 復活(POLICY §6 P0)

**282 already ON 時の追加 rollback 順序**:

1. **Phase A0(先に fetcher 側 preflight gate を OFF)**: `gcloud run services update yoshilover-fetcher --remove-env-vars=ENABLE_GEMINI_PREFLIGHT`
   - 効果:fetcher で preflight skip 自体が発生しない、silent skip risk 消失
2. **Phase A(publish-notice 側 visible 化 OFF)**: §6 Phase A 通り
3. **Phase B/C**: 必要時(image / state cleanup)

**判断 tree**:

```
preflight skip silent risk あり?

└── 282 ON → Phase A0 (fetcher gate OFF) → Phase A (publish-notice visible OFF) 順序
```

このため、282 と 293 は **flag OFF / ON が両方同期** していることが重要。POLICY §7 順序固定はこの整合のためでもある。
