## Phase 1. candidate pickup logic

### direct publish pickup

`scan()` fetches WP posts after the publish-notice cursor and only keeps `status=publish` posts. Those are queued as normal per-post notifications before guarded-review scans run.

- direct publish fetch and enqueue: `src/publish_notice_scanner.py:2713-2749`
- request payload from WP post: `src/publish_notice_scanner.py:1377-1385`

Important detail:

- the direct publish path does **not** force the final mail subject to `【公開済】`
- the final subject is decided later by `send()` after `_classify_mail()`
- therefore a freshly published post can still become `【要確認】` or `【要review】`

Relevant sender path:

- review / publish / x-candidate classification: `src/publish_notice_email_sender.py:1456-1579`
- subject prefix selection: `src/publish_notice_email_sender.py:1373-1378`
- send path and final suppression gate: `src/publish_notice_email_sender.py:2139-2169`

### guarded / review pickup

publish-notice also scans guarded-publish history and emits review-hold notifications from that lane.

- guarded history incremental scan: `src/publish_notice_scanner.py:1029-1071`
- guarded review eligibility: `src/publish_notice_scanner.py:1078-1094`
- guarded review request creation: `src/publish_notice_scanner.py:1774-1887`

This path emits override subjects such as:

- `【要review】`
- `【要確認】`
- `【要確認(古い候補)】`

### publish-only filter behavior

The live filter is narrower in name than in implementation: it suppresses any `mail_type=per_post` whose **final subject** does not start with `【公開済】`.

- env flag read: `src/publish_notice_email_sender.py:1667-1673`
- suppression rule: `src/publish_notice_email_sender.py:1680-1690`

This means:

- summary mail is exempt
- alert / warning / urgent mail is exempt
- per-post mail from the direct publish path is **not** exempt if the final classification changes its subject away from `【公開済】`

## Root cause classification

### stop root cause

**Root cause: filter over-suppression.**

`ENABLE_PUBLISH_ONLY_MAIL_FILTER=1` is currently implemented against the **final subject prefix**, not against the publish source path or original `notice_kind`.

Because of that:

- a post can be freshly published
- publish-notice can pick it up correctly
- `_classify_mail()` can still assign review-like mail class
- the subject becomes `【要確認】` or `【要review】`
- `_should_suppress_publish_only_mail()` then drops it

This is why the job keeps logging:

- `[scan] emitted>0`
- `[summary] sent=0 suppressed=N`
- `[ALERT] publish-notice emitted=N but sent=0`

without any SMTP or runtime error.

### not the root cause

The audit did **not** find evidence for:

- SMTP failure
- recipient resolution failure
- Cloud Run job crash
- summary-path suppression
- alert/warning log suppression
- scanner inactivity

## Recovery condition

### minimum recovery condition

To restore per-post notifications for real published posts, the live filter must stop keying only on `subject.startswith(\"【公開済】\")`.

Any one of the following would recover the path:

1. disable `ENABLE_PUBLISH_ONLY_MAIL_FILTER`
2. keep the flag on, but exempt direct publish scan output even when classification becomes `review`
3. keep the flag on, but suppress only explicit review-only lanes such as guarded-review / post-gen-validate, not all non-`【公開済】` per-post mail

### practical interpretation

If the intended policy is:

- "send mail only for actually published posts, even if they are classified for manual review"

then current live behavior is wrong and needs a live config / code-path change.

If the intended policy is:

- "send mail only when the final subject is `【公開済】`, and suppress all review-like post mail even for published posts"

then current live behavior is operating as coded, and the zero-sent summaries are expected noise rather than a breakage.
