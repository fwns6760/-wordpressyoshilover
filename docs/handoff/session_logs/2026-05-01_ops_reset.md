## 3. 状態定義(永続、必須 evidence)

| 状態 | 意味 | 必須 evidence |
|---|---|---|
| **CODE_DONE** | code 書き終わった | git diff 確認 + local pytest baseline 維持(failures 増加 0) |
| **PUSHED** | origin/master に push | `git log origin/master | grep <hash>` 確認 |
| **DEPLOYED** | image rebuild + service/job update 完了 | `gcloud run describe` で image tag 確認 |
| **LIVE_TRIGGERED** | 自然発火 trigger で 1 回以上 execute | execution log で exit 0 確認 |
| **USER_VISIBLE** | user が確認できる経路に到達 | mail / WP / dashboard 等で sample 観測 |
| **OBSERVED_OK** | user 期待値通りの挙動を 24h or 一定期間観察済 | 24h log breakdown / Gmail sample / silent skip 0 / 既存導線維持 |
| **DONE** | OBSERVED_OK + close 条件達成 | OBSERVED_OK 全項目 + ticket close 判断 |

evidence が無いものは **前段階のまま**(推測で進めない)。

### 状態昇格の禁止 pattern
- `CODE_DONE` → `DONE` への直接昇格(間に PUSHED / DEPLOYED / LIVE_TRIGGERED / USER_VISIBLE / OBSERVED_OK を経由必須)
- `DEPLOYED` 単独で DONE 扱い(LIVE 動作未確認なら NOT DONE)
- env flag OFF deploy を DONE 扱い(効果未測定 → OBSERVE 止まり)

## 7. Acceptance Pack(永続、必須 13 項目、ChatGPT Pro 監査採用版)

user GO が必要な場合、以下 format で **必ず提示**:

```
## Acceptance Pack: <ticket_id>

- Decision: GO / HOLD / REJECT(本 Pack の総評)
- Requested user decision: <user に求める判断 1 行>
- Scope: <変更範囲、touch する file / service / env>
- Not in scope: <意図的に触らないもの、混入禁止>
- Why now: <なぜこの timing で判断するか>
- Preconditions: <着手前提条件(289 24h 安定 / 293 完遂 等)>
- Cost impact: <Gemini call / Cloud Run / mail / 課金影響>
- User-visible impact: <user が体感する変化(publish 数 / mail subject / 件数 等)>
- Rollback: <1 コマンド or revert 手順>
- Evidence: <必要な事前 evidence / 完了 evidence>
- Stop condition: <着手後即停止すべき条件>
- Expiry: <本 Pack の有効期限>
- Recommended decision: GO / HOLD / REJECT
- Recommended reason: <1-2 行>

User reply format: 「GO」/「HOLD」/「REJECT」のみ
```

### 禁止 pattern
- Acceptance Pack なしの「進めてよいですか?」「確認お願い」(**全面禁止**)
- 候補 2-3 並列で user に選ばせる(推奨 1 つに圧縮)
- 影響範囲 / rollback 手順なしの GO 提示
- Cost impact / Stop condition / Expiry の省略
- 不安なら user に聞くのではなく **Acceptance Pack を作る**

### 推奨判断の決め方

- 全 13 項目埋まる + Cost impact ≤ 想定 + Rollback 1 コマンド可能 → **GO 推奨**
- Preconditions 未達 / Cost impact 不明 / Rollback 不能 → **HOLD 推奨**
- Scope に IRREVERSIBLE 含む or 公開影響大 → **REJECT 推奨**(必要に応じて scope 縮小して再 Pack)

## 8. silent skip / cost 増加の検知ルール(永続)

### silent skip = P0(検出方法 4 軸)

1. **ledger silent**: GCS の `*_history.jsonl` で `skip_reason` missing / null / empty な record 検出 → 即 P0 報告
2. **mail 経路 silent**: skip event が log に出ているが publish-notice / mail に乗らない → P0 報告
3. **rss_fetcher 内 silent**: `_log_article_skipped_*` event 出力されたが ledger 永続化されない → P0 報告
4. **WP 直接 silent**: WP に書き込まれた / 削除されたが log / ledger に痕跡なし → P0 報告

### cost 増加検知(Gemini)

1. **24h Gemini call 数監視**: `gemini_call_made=true` count、deploy 前後で +20% 超は user 報告
2. **cache_hit ratio 監視**: 75% を baseline、急激な変動(±15%pt 超)は要因 audit
3. **同 source_url_hash 重複 call 検出**: 1 source per 24h で複数 call は dedup 不全
4. **新 call site 検出**: src/rss_fetcher.py 等の `_request_gemini` call 場所 grep、増加は user GO 必須

### cost 増加検知(Cloud Run / GCS / Logging)

1. **Cloud Build 1 日 N 回上限**: 不必要な build を抑制(294-PROCESS)
2. **Cloud Run min-instance ≠ 0**: idle cost 発生、user GO で見直し
3. **GCS write volume**: ledger 書込頻度の急増 detection
